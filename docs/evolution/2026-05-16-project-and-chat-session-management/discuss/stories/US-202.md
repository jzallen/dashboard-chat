# US-202: Returning user lands in last-used project on sign-in

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (returning, with one or more projects)
> **Job mapped**: J002-Job-1 (Resume)
> **Slice**: Slice 1 — Walking skeleton
> **Anticipated machine state names**: `resolving_initial_scope`,
> `project_selected`, `loading_session_list`, `session_list_visible`

## Problem

Maya has been using Dashboard Chat for a week. She has three
projects ("Q3 Sales", "Q4 Analytics", "Marketing 2026") and uses
"Q4 Analytics" daily. Every morning when she signs in, the app
lands on a bare app shell — she has to click "Projects" in the nav,
scan the grid for "Q4 Analytics", click it, then click into her
most recent session.

That's 3–4 clicks of "the app forgot where I was yesterday." For a
tool she uses daily, the friction compounds — and worse, it
contradicts the J-002 promise that
`active_scope.project_id` is **always** resolved before any
J-002-territory page paints.

## Who

* **Maya Chen (returning)** — completed J-001 yesterday or earlier;
  has one or more projects in her org; opens Dashboard Chat in a
  fresh tab and expects the app to remember which project she was
  most recently active in.

## Solution

J-002 enters `resolving_initial_scope` on transition from J-001
`ready`. It calls `list_projects(user=maya)` AND, for each project,
queries the most-recent session's `last_active_at` via
`list_sessions(project_id, user, limit=1)`. The project with the
globally most-recent session is the **last-used project**. J-002
transitions to `project_selected` for that project, then through
`loading_session_list` to `session_list_visible`. The FE paints the
project chip + the session list on the same first paint.

**Tie-breaker rule** (per the journey YAML): if two projects share
the same `last_active_at`, J-002 picks the project with the
**lexicographically smaller id** (deterministic; never a guess). If
the user has no prior sessions in any project (e.g., they created
projects but never chatted in them), J-002 falls back to the
**lexicographically first project name** (still deterministic).

## Elevator Pitch

* **Before**: Maya signs in → app shell paints with no project →
  she clicks Projects → grid renders → she clicks Q4 Analytics →
  session list renders → she clicks her most recent session. 4
  clicks of "where was I?"
* **After**: Maya signs in → J-002 reads `list_projects` +
  most-recent `last_active_at` per project → transitions to
  `project_selected` for "Q4 Analytics" → app shell paints project
  chip + session list on the same first paint → she clicks her
  most recent session. 1 click of "let me pick up where I left off."
* **Decision enabled**: Maya decides which prior session to resume
  (or whether to start fresh) — the app has narrowed the decision
  space to "what conversation am I continuing today?" rather than
  "what project am I in today?"

## Domain examples

### 1: Happy path — Maya resumes Q4 Analytics

Maya completes J-001 with org "Acme Data". J-002 enters
`resolving_initial_scope`. The projection contains
`projects = ["Q3 Sales", "Q4 Analytics", "Marketing 2026"]` and
`most_recent_session_per_project = {"Q4 Analytics": "T+0", "Q3
Sales": "T-1day", "Marketing 2026": "T-3days"}`. J-002 picks
"Q4 Analytics" (max `last_active_at`). Transitions to
`project_selected` → `loading_session_list` → `session_list_visible`.
Project chip paints "Q4 Analytics" on the same first paint as the
session list.

### 2: Edge — User has projects but no sessions in any

Maya created three projects on Monday and signed out before
chatting. On Tuesday she signs in. `list_sessions` returns empty
for each project. J-002 picks "Q3 Sales" (lexicographically first)
deterministically. The session list is empty
(`no_sessions_empty_state` sub-shape); welcome chips visible.

### 3: Edge — Tie-broken last_active_at

Two projects' most-recent sessions have exact-equal timestamps (an
unlikely but possible case if J-002 wrote sessions in a batch). The
lexicographically smaller project id wins. Documented as part of the
projection-resolution contract.

### 4: Boundary — `list_sessions` transient failure during last-used resolution

J-002 calls `list_sessions` for each project to find the
most-recent. One call fails 503. J-002 has two choices: (a)
transition to `error_recoverable`, OR (b) pick the most-recent
among the successful calls. **Decision (DESIGN-deferred via
OQ-J002-4): we pick option (b)** — partial-result last-used
resolution is preferable to blocking sign-in on a transient backend
hiccup. The failed-project's most-recent is treated as `null`. The
projection emits a `last_used_resolution_degraded` event for
observability.

## Acceptance Criteria

* [ ] On entry from J-001 `ready` with at least one project, J-002
  selects the last-used project (most-recent `session.last_active_at`
  across all projects) and transitions to `project_selected`.
* [ ] If no sessions exist in any project, J-002 falls back to the
  lexicographically first project (deterministic).
* [ ] On a `last_active_at` tie, J-002 picks the lexicographically
  smaller project id (deterministic; never a guess).
* [ ] A transient `list_sessions` failure during last-used resolution
  emits `last_used_resolution_degraded` but DOES NOT block sign-in;
  J-002 falls back to partial-result resolution.
* [ ] The FE app shell paints the project chip on the same first
  paint as the org chip — never one before the other (extends J-001
  K2 invariant).
* [ ] The session list paints on the same first paint as the project
  chip (no flicker; no separate loading-skeleton phase ≥100ms).
* [ ] The TS harness exposes `harness.j002.assert_initial_project(project_name)`
  reading from the same projection the FE consumes.

## Outcome KPIs

* **Who**: Returning users with one or more projects.
* **Does what**: Lands in their last-used project's `session_list_visible`
  state within 800ms of J-001 `ready`, with the project chip and
  session list painting together.
* **By how much**: 100% reach `session_list_visible` at p99; project
  chip first-paint ≤800ms after J-001 `ready` at p95.
* **Measured by**: FE event
  `j002_initial_project_resolved { project_id, latency_ms, degraded: boolean }`;
  correlated with `app_shell_first_paint` from J-001's KPI K2.
* **Baseline**: Today, ~70% of returning users navigate manually
  (developer recall; uninstrumented). First-paint latency for a
  manually-navigated project view is ~2 seconds.

  HAS projects.
* The last-used resolution algorithm needs the projection to carry
  `most_recent_session_per_project` OR the orchestrator's machine
  context to query it on entry. DESIGN owns the read shape (OQ-J002-5).
  — **CLOSED 2026-05-17**: OQ-J002-5 resolved projection-carries,

* Tie-breaker by lexicographic id, not by name, because names may
  change (`update_project`); ids are stable.
