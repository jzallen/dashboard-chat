# US-206: New-session lifecycle (lazy creation; title from first message)

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (starting a fresh conversation)
> **Job mapped**: J002-Job-5 (Start a fresh conversation cleanly)
> **Slice**: Slice 3 — New session lifecycle
> **Anticipated machine state names**: `session_list_visible`
> (`no_sessions_empty_state` sub-shape too), `session_active_no_messages`,
> `session_active`

## Problem

`features/chat-first-ui.feature:36-41` declares: "New Session starts
a fresh conversation: the chat history is empty AND a new session
is created." But "a new session is created" today happens eagerly:
clicking "New Session" creates a session row IMMEDIATELY, even if
Maya never types anything. That produces empty "ghost" sessions in
her session list — clutter that has no semantic value.

`features/chat-first-ui.feature:142-145` simultaneously declares
"session title defaults to first message" — which implies the
session is named (and probably created) AT the first-message
boundary, not at the New-Session click. The two requirements are
in tension under the eager-create shape.

The right model is **lazy creation**: clicking "New Session"
transitions J-002 to `session_active_no_messages` (a welcome state)
with NO session row yet. The session row is created on the first
message, with title set from that message.

## Who

* **Maya Chen (starting a fresh conversation)** — clicks "+ New
  Session" in the nav OR clicks a suggestion chip that leads to a
  new session OR is in the `no_sessions_empty_state` after entering
  a fresh project. Wants the welcome state instantly, the chat
  input ready, and no ghost-session-row created if she navigates
  away without typing.

## Solution

On `new_session_clicked` from any state, J-002 transitions to
`session_active_no_messages`. This is a **purely client-side** state
— no backend call yet. The FE renders:

* The project chip (unchanged).
* The recent-sessions nav rail (unchanged).
* The main pane: a welcome chip layout
  (`features/chat-first-ui.feature:71-76`) — greeting + chips for
  "Upload CSV" and "Browse Projects" — plus an enabled chat input.

`state.session_id` is null in this state. No `POST
/api/projects/{id}/sessions` call has been made yet.

On `first_message_sent` (user types and presses Enter), J-002
eagerly creates the session: `create_session(project_id, user)` per
`backend/app/use_cases/session/create_session.py`. On success, the
new session_id flows into `state.session_id` and J-002 transitions
to `session_active`. The first message is then dispatched to the
agent with the new session_id as `thread_id`.

The session title is set from the first message (truncated to ~80
chars) via `update_session(session_id, {"title":
first_message[:80]})` — fire-and-forget at the J-002 level since
title is a read-model not in the critical path of the chat turn.

## Elevator Pitch

* **Before**: Maya clicks "New Session". A session row is created
  in the backend. She thinks "wait, I'm not sure what I want yet"
  and navigates away. The empty session sits in her list forever
  (or until she manually deletes it).
* **After**: Maya clicks "New Session". She lands on the welcome
  state with the project chip + chips + chat input. She types her
  first message: "Show me top customers by revenue." The session
  is created with that title; she sees the agent's response;
  conversation continues.
* **Decision enabled**: Maya can experiment with "what do I want to
  ask?" before committing to a session. Sessions exist when they
  have content.

## Domain examples

### 1: Happy path — Maya creates a new session in Q4 Analytics

J-002 is in `session_list_visible` for Q4 Analytics. Maya clicks "+
New Session" in the nav rail. J-002 transitions to
`session_active_no_messages` instantly (no backend call). The main
pane re-renders: welcome chips + chat input. Maya types "Show me
top customers by revenue" and presses Enter. J-002 fires
`create_session(q4_id, maya)` and on success transitions to
`session_active`. The session appears at the top of the recent-sessions
nav with title "Show me top customers by revenue". The agent
streams a response with the new session_id as thread_id.

### 2: Edge — Maya navigates away without typing

J-002 is in `session_active_no_messages`. Maya clicks a different
project. J-002 transitions to `switching_project` (per US-207, which
ships the switching_project state contract). No
session row was created in Q4 Analytics; no ghost row sits in
the list.

### 3: Edge — Maya clicks an existing session from `session_active_no_messages`

J-002 is in `session_active_no_messages`. Maya clicks a recent
session in the nav rail. J-002 transitions to `resuming_session`
(per US-205). No session row was created in `session_active_no_messages`;
the recent-sessions list is unchanged.

### 4: Edge — `create_session` fails transiently on first-message-sent

Maya types her first message and presses Enter. J-002 fires
`create_session`. Backend returns 503. J-002 transitions to
`error_recoverable` with copy "we couldn't start the session — try
again." Maya clicks "Try again"; J-002 re-enters
`session_active_no_messages` with the typed message PRESERVED in the
composer (the user does not retype). She presses Enter again; the
retry succeeds.

### 5: Edge — User starts typing then sends with a single character

`features/chat-first-ui.feature:142-145` says "session title defaults
to first message." If the first message is a single character "a",
the title is "a". J-002 does not add ceremony; the title can be
edited later via `update_session` per
`features/chat-first-ui.feature:147-150`.

## Acceptance Criteria

* [ ] On `new_session_clicked`, J-002 transitions to
  `session_active_no_messages` without firing any backend call;
  `state.session_id` is null.
* [ ] The `session_active_no_messages` UI renders the welcome
  chips ("Upload CSV", "Browse Projects") and an enabled chat
  input; the project chip is preserved from the prior state.
* [ ] On `first_message_sent`, J-002 fires
  `create_session(project_id, user)` and on success transitions to
  `session_active` with `state.session_id = response.session_id`.
* [ ] On success, J-002 fires (fire-and-forget) `update_session(
  session_id, {title: first_message[:80]})` so the session title
  appears in the recent-sessions nav within a second.
* [ ] On `create_session` transient failure, J-002 transitions to
  `error_recoverable` with the originating `correlation_id`; the
  composer text is preserved across the retry boundary.
* [ ] Navigating away from `session_active_no_messages` to ANY other
  state without sending a first message leaves NO session row in
  the backend.
* [ ] Clicking an existing session from `session_active_no_messages`
  transitions to `resuming_session` (and the new-session intent is
  cancelled).
* [ ] The TS harness exposes `harness.j002.start_new_session()` and
  `harness.j002.send_first_message(content)`.

## Outcome KPIs

* **Who**: Users starting a fresh conversation via the New Session
  affordance.
* **Does what**: Lands in `session_active_no_messages` instantly
  (no backend round-trip); creates a session row only on first
  message.
* **By how much**: 100% of New-Session clicks paint the welcome
  state at p95 ≤150ms (no backend latency); 0 ghost session rows
  per instrumented session-day (an empty session row is a
  contract violation, not a metric to optimize).
* **Measured by**: FE events
  `j002_new_session_welcome_painted { project_id, latency_ms }`,
  `j002_first_message_sent_session_created { project_id, session_id,
  latency_ms }`. Backend acceptance test: assert no session row
  exists for an `session_active_no_messages → switching_project`
  trace.
* **Baseline**: Today, every New-Session click creates a session
  row eagerly. The ghost-row rate is 100% for sessions where the
  user navigated away without typing (anecdotally 10-20% of new
  sessions).

## Technical notes

* Depends on US-203 (session_list_visible).
* The lazy-create pattern requires that the chat-turn dispatch
  (POST /chat) can carry `thread_id = null` for the FIRST turn and
  then bind it to the newly-created session_id mid-stream OR can
  fail-fast if the session_id is not yet known. The simplest
  implementation: J-002 awaits `create_session` BEFORE dispatching
  the first chat turn — the user perceives this as a single
  Enter-keystroke → response (the create_session latency is
  ~50-100ms p95, well within the "first message" UX budget).
* Title editing is in scope at the per-row level
  (`features/chat-first-ui.feature:147-150` already covered by
  existing `update_session`) — not in scope for this story since
  it's single-step CRUD per anti-pattern 1.
