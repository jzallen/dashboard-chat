# US-208: Chat-agent invocation carries `active_scope` from J-002's projection (agent rejects missing scope)

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (sending any chat turn) + any developer
> writing a test or curl-based integration against the agent
> **Job mapped**: J002-Job-6 (Stay scoped during a chat turn)
> **Slice**: Slice 4 — Project switching + agent contract
> **Anticipated machine state names**: applies to any state where
> the chat input is enabled; primarily `session_active` and
> `session_active_no_messages`

## Problem

`agent/lib/chat/handleChat.ts:75` today destructures `project_id`
from the request **body**. Any client (the FE, a curl, a malformed
headless harness call) can send `project_id` set to a value
unrelated to the JWT's `org_id` claim. There is no validation. A
chat turn submitted with `project_id = "project-of-a-different-org"`
will be processed by the agent against that project's context — a
cross-tenant data-leak surface.

J-001 introduced ADR-029 §4 as a forward-looking contract: every
chat-agent invocation carries `X-Active-Scope` from the FE's
projection; agent rejects turns missing `org_id` or `project_id`
with 400. But J-001 has no chat surface in its `ready` state — the
contract was specified but not exercised. **J-002 is the first
flow that exercises the contract end-to-end** because J-002 is the
first machine to dispatch chat turns post-J-001.

This story is the load-bearing implementation of that contract. It
landing during J-002's DELIVER is what retires the cross-tenant
data-leak surface for ALL chat turns (J-002's, and every future
J-003+ flow's chat-shaped sub-states).

## Who

* **Maya Chen (any user sending a chat turn)** — wants the agent
  to operate on the project the FE shell says she's in, not a
  different project from a stale or malformed body field.
* **Developer writing acceptance tests** — wants a clean contract
  to assert against: every chat turn carries scope; missing-scope
  turns are 400.
* **Security-minded operator** — wants the cross-tenant data-leak
  surface mechanically closed.

## Solution

Two layers of work:

1. **FE / loader side**: the Remix loader's `uiStateClient` helper
   (per ADR-029 §2 Option D) sets the `X-Active-Scope` header on
   ALL outgoing fetch calls to the agent (and to the backend). The
   header value is JSON-encoded from `active_scope` read out of the
   J-002 projection. No FE component sets this header manually;
   the helper is the only writer.

2. **Agent middleware side**: `agent/lib/chat/handleChat.ts` (and
   any future chat-shaped endpoints) read scope from the
   `X-Active-Scope` header EXCLUSIVELY. The body's `project_id`
   field is ignored (and, in a follow-up cleanup, removed from the
   request schema). Missing or malformed `X-Active-Scope` → 400
   with the body
   `{"error": "agent invocation missing scope: missing <field>"}`.

The auth-proxy is the trust boundary — it forwards the header
verbatim from the FE to the agent (no rewrites). The agent trusts
the header value AND validates `org_id` matches the
auth-proxy-injected `X-Org-Id` header (defense in depth — but
J-002's contract is that the FE never sends mismatched scope; the
agent validation is a safety net, not the primary defense).

## Elevator Pitch

* **Before**: `agent/lib/chat/handleChat.ts:75` reads `project_id`
  from the request body with no validation. A malformed client can
  poison the agent's context.
* **After**: The agent reads `org_id` and `project_id` from
  `X-Active-Scope` header, set by the Remix loader from J-002's
  projection. Missing or mismatched → 400. The body's `project_id`
  field is ignored.
* **Decision enabled (Maya)**: Maya trusts that "the agent is
  working on the project I see in the chip." Cross-tenant data
  leakage is mechanically impossible.
* **Decision enabled (developer)**: Acceptance tests assert
  `X-Active-Scope` presence on every agent call AS A CONTRACT,
  not as a heuristic.

## Domain examples

### 1: Happy path — chat turn from `session_active`

J-002 is in `session_active` for project Q4 Analytics with session
chat-9b2a. Maya types "show me top customers" and presses Enter.
The Remix loader's `uiStateClient` helper composes the POST to
`/chat`. The outgoing request carries:

```
POST /chat HTTP/1.1
Authorization: Bearer <jwt>
X-Active-Scope: {"org_id":"acme-data-abc123","project_id":"q4-analytics-def456","resource_type":null,"resource_id":null}
X-Correlation-Id: R-chat-7y8c
Content-Type: application/json

{"messages":[{"role":"user","content":"show me top customers"}],"thread_id":"chat-9b2a"}
```

The agent middleware reads `X-Active-Scope`, parses it, validates
both `org_id` and `project_id` are present (✓), validates
`org_id` matches `X-Org-Id` (✓), and proceeds. The body's optional
`project_id` field is NOT read.

### 2: Edge — chat turn from `session_active` with a dataset attached

Same as Example 1 but `session_active` carries `active_scope.resource_*`
set. The header is:

```
X-Active-Scope: {"org_id":"acme-data-abc123","project_id":"q4-analytics-def456","resource_type":"dataset","resource_id":"sales_2026-xyz"}
```

The agent uses `resource_id` to inject dataset-context into the
prompt (per `agent/lib/chat/handleChat.ts:113-115`).

### 3: Failure — malformed client sends no header

A curl call to `POST /chat` omits `X-Active-Scope` entirely. The
agent middleware returns 400 with body
`{"error": "agent invocation missing scope: missing org_id"}`. No
LLM call is made.

### 4: Failure — client sends header but org_id is missing

`X-Active-Scope: {"project_id":"q4-..."}` (no org_id). 400 with
body `{"error": "agent invocation missing scope: missing org_id"}`.

### 5: Failure — header's org_id mismatches JWT claim

`X-Active-Scope: {"org_id":"other-org","project_id":"q4-..."}` but
the JWT's `org_id` claim is `acme-data-abc123`. The auth-proxy
injected `X-Org-Id: acme-data-abc123`. The agent middleware compares
and returns 403 with body
`{"error": "scope mismatch: jwt.org_id != X-Active-Scope.org_id"}`.

### 6: Backwards-compat migration window

For the migration period (one release per technical-notes), the
agent middleware reads `X-Active-Scope` PREFERENTIALLY, falls back
to the body's `project_id` for backwards-compat, and emits a
`scope_header_fallback_used` log event. After the migration window
closes, the body's `project_id` is removed from the request schema
and the fallback is deleted.

## Acceptance Criteria

* [ ] The Remix loader's `uiStateClient` helper (per ADR-029 §2)
  sets `X-Active-Scope` on every outgoing fetch to the agent and
  to the backend; FE chat-view components do NOT set this header
  themselves.
* [ ] The agent middleware (`agent/lib/chat/handleChat.ts` and any
  future chat-shaped endpoints) reads `org_id` and `project_id`
  from `X-Active-Scope` EXCLUSIVELY (after the migration window).
* [ ] Missing `org_id` in `X-Active-Scope` → 400 with body
  `{"error":"agent invocation missing scope: missing org_id"}`.
* [ ] Missing `project_id` in `X-Active-Scope` → 400 with body
  `{"error":"agent invocation missing scope: missing project_id"}`.
* [ ] `X-Active-Scope.org_id != X-Org-Id` (auth-proxy-injected) →
  403 with body
  `{"error":"scope mismatch: jwt.org_id != X-Active-Scope.org_id"}`.
* [ ] During the migration window (one release post-DELIVER), the
  agent falls back to the body's `project_id` and emits a
  `scope_header_fallback_used` log event with the calling client
  identifier. After the window closes, the fallback is removed
  and the body's `project_id` field is dropped from the request
  schema.
* [ ] The TS harness exposes
  `harness.j002.assert_agent_received_scope(turn_index)` reading
  the captured outgoing request headers.

## Outcome KPIs

* **Who**: Every user sending a chat turn; every test/curl call to
  the agent.
* **Does what**: Every chat turn carries scope; missing-scope turns
  are rejected with a named diagnostic.
* **By how much**: 100% of FE-originated turns carry scope at p99;

* **Baseline**: Today, the agent has zero scope validation
  (`agent/lib/chat/handleChat.ts:75` reads `project_id` from body
  unconditionally). The cross-tenant data-leak surface is open by
  default.

## Technical notes

* Depends on the Remix loader pattern being live (US-202's
  walking-skeleton path landed `app/root.tsx` reading the J-001
  projection; this story extends `uiStateClient` to set the
  header on outgoing calls).
* The migration-window flag (`SCOPE_HEADER_FALLBACK_ENABLED`) is a
  one-release temporary; after the window closes, the flag and the
  fallback both delete in the same change. **Concrete cutover
  contract** (per R8 in handoff-design.md):
  - The flag's value is hard-coded to the J-002 DELIVER release
    version PLUS one release.
  - `agent/lib/chat/handleChat.ts` includes a compile-time check
    (a const assertion against the current package version) that
    fails the build if the sunset date has passed AND the flag is
    still in the codebase. This forces the flag-removal change to
    land on time.
  - Migration tracking is via `scope_header_fallback_used { calling_client }`
    log events; the on-call alert (per outcome-kpis.md §5) catches
    legacy clients that never upgrade.
* `agent/lib/chat/handleChat.ts:113-117` shows the dispatch
  context's `projectId` is set from `project_id` (body field). The
  refactor: `projectId = headerScope.project_id`. The
  `contextType` derivation logic stays the same (dataset → dataset,
  null → project per line 117).
* This story does NOT cover the backend's scope-header reading.
  The backend has its own contract — out of J-002's scope; tracked
  for a future cross-cutting refactor.
