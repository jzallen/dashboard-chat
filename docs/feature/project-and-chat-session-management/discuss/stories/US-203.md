# US-203: Project's session list renders sorted by recency on project entry

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (returning, in a project with multiple sessions)
> **Job mapped**: J002-Job-1 (Resume), J002-Job-5 (Start fresh; the
> welcome chips visible in `no_sessions_empty_state` substate)
> **Slice**: Slice 2 — Session list + resume
> **Anticipated machine state names**: `loading_session_list`,
> `session_list_visible`, `no_sessions_empty_state`

## Problem

The chat-first-ui.feature at `features/chat-first-ui.feature:60`
declares: "sessions are sorted by most recent first." It also
declares at lines 64-68 that "recent sessions are displayed at the
bottom of the nav" — capped at 5, each showing "a truncated first
message as the label." But today these requirements are scattered
across the FE's `RecentSessionsList` component, the `SessionsApi.list`
call from `DatasetLayerHarness`, and the React Context that holds
the currently-selected session — three places that can drift.

For a returning user navigating into a project with 10+ sessions,
the session list IS the navigation surface. If the sort is wrong,
or if the list goes stale after a session creation in another tab,
or if the dataset chip on the recent-sessions nav lags the project
switch, Maya feels the inconsistency.

J-002 owns this. The session list is part of the projection that the
ScopeResolver resolves; it paints from a single source.

## Who

* **Maya Chen (returning, in a project with sessions)** — has just
  entered a project (via US-202 last-used resolution, or via direct
  navigation per US-207) and wants to see her sessions ready, sorted
  by recency, with the most-recently-active session clearly first.
* **Maya (in a project with zero sessions)** — sees the
  `no_sessions_empty_state` sub-shape: welcome chips ("Upload CSV",
  "Browse Projects") plus a fresh chat input.

## Solution

On entry to `project_selected`, J-002 fires a side-effect into
`loading_session_list`. The backend call is
`list_sessions(project_id, user, cursor=null, page_size=30)` per
`backend/app/use_cases/session/list_sessions.py:18-58`. The
response carries items sorted server-side by `last_active_at DESC`.
J-002's projection includes `session.list = response.items` and
`session.list_next_cursor`; the FE renders the first page in the
recent-sessions nav (capped at 5) AND on the Chats page (`/sessions`
per `frontend/app/routes.ts:33`).

If `response.items` is empty, J-002's `session_list_visible` state
takes the `no_sessions_empty_state` shape — welcome chips visible,
chat input enabled for `session_active_no_messages` (see US-206).

## Elevator Pitch

* **Before**: Maya enters Q4 Analytics. The session list renders
  (sometimes immediately, sometimes after a delay if the FE re-fetches),
  sorted correctly per the API but with no visible coherence
  between the nav's "5 most recent" and the Chats page's full list.
  When she creates a session in another tab, this tab's list is
  stale until refresh.
* **After**: Maya enters Q4 Analytics. The session list paints with
  the project chip on the same first paint, sorted most-recent-first.
  The nav shows the first 5; clicking "Chats" navigates to the
  full list (also from J-002's projection). When a session is
  created in another tab, J-002's projection-stream picks it up and
  the list refreshes within a second.
* **Decision enabled**: Maya decides which session to resume by
  scanning a list whose ordering she trusts. She never wonders "is
  this list current?"

## Domain examples

### 1: Happy path — Q4 Analytics has 4 prior sessions

Maya enters Q4 Analytics. J-002 calls `list_sessions(q4_id, maya,
limit=30)`. The response is 4 items sorted T4 > T3 > T2 > T1. The
recent-sessions nav renders the 4 with the most-recent-message
preview as label. The Chats page (when Maya navigates to it) shows
the same 4 in the same order.

### 2: Edge — Project has zero sessions

Maya enters a newly-created project (US-201 path). `list_sessions`
returns `{items: []}`. J-002 enters `no_sessions_empty_state`. The
recent-sessions nav shows an empty rail with a "+ New Session" CTA.
The main pane shows the welcome state: "What would you like to
do?" with chips for Upload CSV and Browse Projects.

### 3: Edge — Project has more than 30 sessions

Maya enters a long-running project with 50 sessions. The first page
(30) is rendered; the nav shows the first 5 of the first page (the
most-recent 5). The Chats page shows the full first page with a
"Load more" CTA backed by `list_sessions(..., cursor=next_cursor)`.

### 4: Boundary — Session created in another tab mid-render

Maya has the project's session list visible. She creates a new
session in Tab B. J-002's projection-stream (`GET
/api/flows/project-session-mgmt:maya/projection/stream` per
ADR-027 §1) pushes an updated session list. Tab A's list refreshes
within 1 second, with the new session at the top.

## Acceptance Criteria

* [ ] On entry to `project_selected`, J-002 fires
  `list_sessions(project_id, user, page_size=30)` and transitions
  through `loading_session_list` to `session_list_visible`.
* [ ] The session list is server-side sorted by `last_active_at DESC`;
  J-002 does NOT re-sort client-side.
* [ ] The recent-sessions nav rail renders the first 5 items;
  the Chats page renders the first 30 with pagination available.
* [ ] If `response.items` is empty, the `no_sessions_empty_state`
  sub-shape is entered: welcome chips ("Upload CSV", "Browse
  Projects") visible; recent-sessions rail empty; chat input
  enabled (for `session_active_no_messages` per US-206).
* [ ] The session list and the project chip paint on the same
  first paint (no separate loading skeleton ≥100ms for the list).
* [ ] A session created in another tab refreshes the list within
  1 second via the projection stream (`ADR-027 §1
  GET /api/flows/.../projection/stream`).
* [ ] The TS harness exposes `harness.j002.get_session_list()`
  returning the same items the FE renders.

## Outcome KPIs

* **Who**: Returning users entering a project with sessions.
* **Does what**: Sees the session list painted on first paint,
  sorted most-recent-first, with the project chip.
* **By how much**: 100% of project entries paint list+chip on
  the same first paint at p99; cross-tab refresh ≤1 second at p95.
* **Measured by**: FE event
  `session_list_first_paint { project_id, session_count, latency_ms }`;
  cross-tab refresh event
  `session_list_refreshed_via_projection_stream`.
* **Baseline**: Today, ~85% of project entries paint list+chip
  together (the 15% delta is the project-context race the feature
  retires). Cross-tab refresh is unsupported (developer must
  hard-refresh).

## Technical notes

* Depends on US-202 (or US-201 + US-201's creating_project happy path)
