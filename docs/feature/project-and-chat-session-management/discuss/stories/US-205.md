# US-205: Resuming a session restores transcript and dataset context

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (returning, resuming a specific session)
> **Job mapped**: J002-Job-4 (Resume with dataset context restored)
> **Slice**: Slice 2 — Session list + resume
> **Anticipated machine state names**: `session_list_visible`,
> `resuming_session`, `session_active`

## Problem

`features/chat-first-ui.feature:109-113` commits to: "Given the user
clicks a session in the nav or Chats view, then the ChatView loads
with that session's message history AND the dataset context (if any)
is restored from session metadata." But today,
`backend/app/use_cases/session/update_session.py:50-52` allows
updating only `title` and `last_active_at` — there is no
`active_dataset_id` field on the session row. The dataset-context
restoration is documented but unimplemented.

When Maya resumes a session, her experience is: transcript loads,
but she has to re-attach the dataset by name ("the patients table")
to ask a follow-up question. The conversational continuity breaks at
the resume boundary.

J-002 retires this. The session's last dataset attachment is
persisted (storage shape DESIGN-deferred per D11 / OQ-J002-1); on
resume, J-002 reads it and materializes `active_scope.resource_*`
before transitioning to `session_active`.

## Who

* **Maya Chen (returning, resuming a session)** — clicks a session
  in the recent-sessions nav or the Chats page. Wants the
  transcript AND the dataset context restored. Conversation
  continues without re-attaching.

## Solution

On `session_clicked` from `session_list_visible`, J-002 transitions
to `resuming_session`. It performs two side-effect calls:

1. **Transcript load** — `list_session_events(session_id, user)`
   per `backend/app/use_cases/session/list_session_events.py`. The
   reader is `noop_session_event_reader` today; a future Stream.io
   adapter will materialize the persisted DomainEvents.
2. **Dataset context restore** — reads `session.active_dataset_id`
   from session metadata (storage shape DESIGN-deferred per
   OQ-J002-1: column on session row, side-log, or denormalization
   from event stream).

Both writes are materialized before transitioning to
`session_active`. If the stored dataset id no longer resolves
(deleted), `active_scope.resource_*` is set to null and the chat
input renders an empty-state dataset chip with copy "the dataset
for this session is no longer available." The transcript still
renders; the conversation can continue in conversational mode.

## Elevator Pitch

* **Before**: Maya clicks her "what's avg rev by region" session.
  The transcript loads. She types a follow-up: "now filter to top
  10." The agent responds "what's a column? You have no dataset
  attached." Maya groans and re-types "filter the sales_2026 table
  to top 10."
* **After**: Maya clicks her "what's avg rev by region" session.
  The transcript loads. The dataset chip in the chat input gutter
  reads "sales_2026". She types "now filter to top 10." The agent
  has `active_scope.resource_id` populated and runs the filter
  immediately.
* **Decision enabled**: Maya picks up the conversation where she
  left off — including the dataset context — without re-establishing
  what's already established.

## Domain examples

### 1: Happy path — resume "chat-9b2a" with stored dataset

Maya clicks "chat-9b2a" in the recent-sessions nav. J-002
transitions to `resuming_session`. Two calls fire in parallel:
`list_session_events(chat-9b2a)` returns 12 prior messages;
session metadata returns `active_dataset_id = "sales_2026"`. J-002
materializes both and transitions to `session_active`. The transcript
renders with 12 messages; the dataset chip reads "sales_2026"; the
chat input is enabled.

### 2: Edge — session has no stored dataset_id

Maya resumes a session she opened but never attached a dataset to.
`session.active_dataset_id` is null. J-002 transitions to
`session_active` with `active_scope.resource_*` null; the dataset
chip is absent from the gutter (or shows a "+ attach dataset" CTA);
the chat input is enabled in conversational mode.

### 3: Edge — stored dataset has been deleted

Maya resumes "chat-9b2a". `session.active_dataset_id` is
"sales_2026" but `get_dataset("sales_2026")` returns 404. J-002
transitions to `session_active` with `active_scope.resource_*` null
AND emits a `session_dataset_unavailable` FlowEvent. The dataset
chip renders the empty-state copy. Transcript still renders.

### 4: Edge — stored dataset belongs to a different project

(Anomaly case — shouldn't happen but the contract handles it.)
Session has `active_dataset_id = D` but D belongs to a different
project than the session's `project_id`. ScopeResolver's invariant
4 catches it: session_active is entered with `resource_*` null and a
`scope_reconciled` FlowEvent is emitted for observability.

### 5: Cross-flow — token expires during resuming_session

(Covered by US-209.) Resume kicks off; token expires mid-load;
J-002 transitions to `freeze`; silent re-auth completes; J-002
re-enters `resuming_session` with the same correlation_id; resume
completes.

## Acceptance Criteria

* [ ] On `session_clicked` from `session_list_visible`, J-002
  transitions to `resuming_session` and fires both transcript-load
  and dataset-context-restore calls.
* [ ] Both writes (transcript + `active_scope.resource_*`) are
  materialized before transitioning to `session_active`; no
  half-loaded state is observable to the FE.
* [ ] If stored `active_dataset_id` resolves successfully,
  `active_scope.resource_type = "dataset"` and `resource_id = <id>`.
* [ ] If stored `active_dataset_id` is null OR cannot be resolved
  (deleted, cross-project, access revoked), `active_scope.resource_*`
  is null AND a `session_dataset_unavailable` (deleted) or
  `scope_reconciled` (cross-project / access-revoked) FlowEvent is
  emitted.
* [ ] If `session_id` cannot be resolved (session deleted in
  another tab), J-002 transitions silently back to
  `session_list_visible`; no error panel.
* [ ] The transcript and dataset chip paint on the same first paint
  (no separate skeleton phase ≥100ms for either).
* [ ] The TS harness exposes `harness.j002.resume_session(session_id)`
  and `harness.j002.get_transcript(session_id)`.
* [ ] The session metadata schema for `active_dataset_id` is
  flagged as DESIGN-deferred via OQ-J002-1 in the handoff document
  (this story commits to the *behavior*, not the storage shape).

## Outcome KPIs

* **Who**: Returning users resuming a session with a previously-attached
  dataset.
* **Does what**: Sees the transcript AND the correct dataset chip
  on first paint; types a follow-up without re-attaching the
  dataset.
* **By how much**: ≥95% of session resumes with a stored dataset
  paint the dataset chip correctly on first paint at p95; zero
  re-attachments observed in instrumented sessions (assuming the
  dataset still exists).
* **Measured by**: FE events
  `session_resume_completed { session_id, dataset_present,
  latency_ms }` and `session_dataset_unavailable` for the
  degraded path.
* **Baseline**: Today, 100% of session resumes lose the dataset
  context (it is uninstrumented because it has not been built).

## Technical notes

* Depends on US-203 (session_list_visible).
* Depends on OQ-J002-1 (session-metadata storage shape) being
  resolved in DESIGN before DELIVER.
* Transcript load reads from `list_session_events`; today's reader
  is noop. The story is shippable even with the noop reader if the
  transcript stays empty — the dataset chip is the more visible
  part. If a real Stream.io adapter is required for transcript
  load, that's a backend dependency tracked in OQ-J002-1.
