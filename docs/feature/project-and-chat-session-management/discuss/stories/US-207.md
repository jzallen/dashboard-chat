# US-207: User switches projects within an org — scope atomically retargets

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (in `session_active`, switching to another project)
> **Job mapped**: J002-Job-2 (Switch projects without bleeding context)
> **Slice**: Slice 4 — Project switching + agent contract
> **Anticipated machine state names**: `session_active`,
> `switching_project`, `project_selected`, `loading_session_list`,
> `session_list_visible`

## Problem

The canonical bug class named at `adr-027:14` is the ChatView
project-context race: Maya clicks "Q3 Sales" while she's mid-session
in "Q4 Analytics." For a brief window (200-500ms today), the FE has
inconsistent state — the chip might say "Q3 Sales" while the session
list still shows Q4's sessions, or the chat input might dispatch a
turn with Q4's session_id but a body containing a prompt about Q3
data. Both directions of the race are observable and erode trust.

J-002 retires this. A project switch is a single state transition
(`switching_project`) that atomically invalidates `session_id` and
`resource_*`, retargets `active_scope.project_id`, and re-enters
the project-loading sequence. There is no window where the chip and
the session list disagree.

This is also the path most prone to drift in the existing FE because
TanStack Query caches per-project session lists separately; an
incomplete cache invalidation can re-render the old list briefly
during the switch.

## Who

* **Maya Chen (in `session_active`)** — has been chatting in Q4
  Analytics; clicks "Q3 Sales" in the nav rail OR clicks a Q3 Sales
  project tile from the Projects grid OR pastes a Q3 Sales deep link.
  Wants the project chip + session list + chat surface to all
  retarget on the SAME paint, with no in-flight chat-turn landing
  at the agent under the wrong project.

## Solution

On `switching_project_intent` from any J-002 state, J-002
transitions to `switching_project`. The state's behavior:

1. **Invalidate** `state.session_id` to null AND `active_scope.resource_*`
   to null. The chat-view component unmounts; any in-flight chat turn
   is cancelled at the FE boundary (the SSE stream is closed before
   the next turn is dispatched).
2. **Retarget** `active_scope.project_id` to the new project.
3. **Re-enter** the loading sequence: `switching_project →
   project_selected → loading_session_list → session_list_visible`.
4. **Cancel any pending intents** in the orchestrator's queue that
   were bound to the old project (e.g., a user-clicked `session_clicked`
   that was queued during freeze is dropped if its target session
   belongs to the old project).

The FE's Remix route handles this by re-running the project-route
loader on URL change; the loader reads from J-002's projection with
the new `intent_project_id`. Because the projection is the source
of truth, the chip and session list always come from the same JSON
response — they cannot disagree.

## Elevator Pitch

* **Before**: Maya clicks "Q3 Sales" while mid-chat in Q4. Project
  chip updates → session list briefly still shows Q4 sessions →
  chat input briefly bound to Q4's session_id → session list
  refreshes to Q3 → chat input refreshes. ~300-500ms of "did I
  click the wrong project?"
* **After**: Maya clicks "Q3 Sales". J-002 transitions through
  `switching_project` to `project_selected` for Q3. The FE
  re-renders: chip + session list + chat surface, all from the
  same projection, all on the same paint. <300ms; no flicker; no
  cross-project turn lands at the agent.
* **Decision enabled**: Maya navigates between projects fluidly —
  she trusts the chip names the project the rest of the UI is
  actually rendering against.

## Domain examples

### 1: Happy path — Maya switches from Q4 to Q3 in the nav

J-002 is in `session_active` for Q4 Analytics with session
"chat-9b2a". Maya clicks "Q3 Sales" in the nav rail. J-002
transitions:
* `session_active` → `switching_project` (with new_project_id =
  q3_sales_id; old_project_id = q4_analytics_id)
* The chat-view component unmounts; any in-flight chat turn for
  chat-9b2a is cancelled (the FE's SSE stream is closed).
* `state.session_id = null`; `active_scope.resource_* = null`.
* `active_scope.project_id = q3_sales_id`.
* `switching_project` → `project_selected` → `loading_session_list`
  → `session_list_visible` for Q3 Sales.

Total wall-clock: <300ms at p95.

### 2: Happy path — Maya pastes a Q3 deep link while in Q4

Maya is in Q4. She pastes `/projects/q3-sales` into the address bar
and presses Enter. The Remix loader for the new URL runs; J-002's
`switching_project_intent` event is emitted via the loader's call
to the projection with `intent_project_id = q3_sales_id`. Same
transition path as Example 1.

### 3: Edge — chat-turn in flight at the moment of switch

Maya types "what's the avg rev by region" in Q4 and presses Enter.
The chat turn streams via SSE. 200ms in, while the response is
still streaming, Maya clicks "Q3 Sales". The FE closes the SSE
stream (cancelling the turn at the agent boundary). The chat-view
unmounts. The agent receives no further frames from the closed
stream. Q3 Sales's session list renders.

**Critical**: the in-flight turn does NOT land at the agent with
the Q3 Sales project_id AND the Q4 session_id mixed. Cancellation
is preferred over re-targeting; partial turns are dropped.

### 4: Edge — switch to scope_mismatch_terminal

Maya clicks a project she no longer has access to (membership
revoked since her last session). J-002 transitions
`switching_project → scope_mismatch_terminal` (ScopeResolver
invariant 4 caught the cross-tenant case). Scene 6 panel renders.

### 5: Cross-flow — token expiry during switch

(Covered by US-210.) Switch initiates; token expires mid-load;
J-002 transitions to `freeze`; silent re-auth completes; J-002
re-enters `switching_project` with the original correlation_id;
switch completes.

## Acceptance Criteria

* [ ] On `switching_project_intent`, J-002 transitions to
  `switching_project` and invalidates `state.session_id` to null
  AND `active_scope.resource_*` to null BEFORE any new project-loading
  call fires.
* [ ] Any in-flight SSE chat-turn stream from the OLD project is
  closed at the FE boundary before the new project's loader runs;
  the agent receives no further frames from that stream.
* [ ] The agent NEVER receives a chat-turn POST carrying both the
  old project's session_id and the new project's project_id in the
  same request (the cancellation contract from above prevents this).
* [ ] On successful switch, J-002 reaches `session_list_visible` for
  the new project within 300ms at p95.
* [ ] The project chip and the session list for the new project
  paint on the same first paint; no Q4 session appears in Q3's list
  at any point during the switch.
* [ ] On cross-tenant or revoked-access switch, J-002 transitions
  `switching_project → scope_mismatch_terminal` (NOT
  `project_selected` followed by an error); the named-diagnostic
  panel renders.
* [ ] **Cache invalidation invariant (R9 from handoff)**: Before
  `loading_session_list` fires for the new project, the FE's
  TanStack Query cache for `list_sessions(old_project_id)` is
  invalidated OR refetched-on-stale. No race window exists where
  the new project's `session_list_visible` could render the old
  project's cached list. The Remix loader pattern naturally
  achieves this (loader re-run replaces the route data); the AC is
  documented here so the implementation team treats it as a
  contract, not an implementation detail.
* [ ] The TS harness exposes `harness.j002.switch_project(project_name_or_id)`.

## Outcome KPIs

* **Who**: Users switching projects within their org (any path:
  nav click, deep link, Projects grid).
* **Does what**: Lands in the new project's `session_list_visible`
  state with chip + session list paint together, in <300ms p95;
  zero observed chat-turns landing at the agent under the wrong
  project.
* **By how much**: 100% atomic at p99 (chip and list paint
  together; no cross-project session in the list); switch latency
  ≤300ms at p95; cross-project chat-turn rate at 0 (a single
  observed violation is a bug, not a metric to optimize).
* **Measured by**: FE event
  `j002_project_switch_completed { old_project_id, new_project_id,
  latency_ms, in_flight_turn_cancelled: boolean }`; backend
  acceptance test: assert no agent request carries mismatched
  project_id/session_id pairing.
* **Baseline**: Today, ~10-20% of project switches observe a brief
  flicker (developer recall; the ChatView project-context race).
  Cross-project chat-turn rate is uninstrumented but is known to
  occur during rapid-switch scenarios.

## Technical notes

* Depends on US-202 (last-used resolution AND `project_selected` /
  `session_list_visible` / `session_active` state contracts already
  in place).
* The SSE cancellation contract requires the FE's ChatView
  component to call `eventSource.close()` on its `useEffect` cleanup
  when `state.session_id` invalidates. ADR-027 §1 mentions
  `RedisFlowEventLog` for projection streams; the chat agent's SSE
  stream is a separate concern but the same cancellation pattern
  applies.
* The orchestrator's replay buffer (ADR-027 §5) should NOT replay
  intents from the old project on `THAW` if those intents are
  bound to a session_id that no longer exists in the new project
  context — DESIGN owns the filtering rule (OQ-J002-6 in handoff).
