# US-209: Dataset context switching via agent's resolve_dataset OR direct selection

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (in `session_active`, referencing a dataset
> by name in a chat turn OR picking a dataset directly)
> **Job mapped**: J002-Job-7 (Switch dataset context inside a session)
> **Slice**: Slice 5 — Dataset context switching
> **Anticipated machine state names**: `session_active`,
> `switching_dataset_context`

## Problem

`features/chat-first-ui.feature:89-93` declares: "Given no dataset
is selected as context, when the user issues a table operation
command (e.g., 'filter rows where age > 30'), then the chat displays
an inline list of available datasets and the user can click a
dataset to select it as context, and the command is re-processed
with the selected dataset."

This is implemented today as:
1. `agent/lib/chat/tools.ts:13-22` exposes `resolve_dataset` as a
   conversational tool.
2. `agent/lib/chat/handleChat.ts:99-104` intercepts the
   `tool-input-available` chunk via `pipeChatStream` and emits a
   `data-agent-request` typed part.
3. The FE consumes the typed part, shows an inline dataset list,
   and on user pick re-submits the chat turn with `contextType
   = "dataset"` and `contextId = <picked-dataset-id>`.

But the FE state for "what dataset is currently in scope" lives in
a TanStack Query key + a route param + a React Context — three
places that can drift. The FE is essentially carrying a parallel
state machine that J-002 should own.

J-002 retires this. The agent's `resolve_dataset` tool-call return
becomes a `dataset_resolved_by_agent` EVENT to J-002. Direct
selection (clicking a dataset from a list) becomes a
`dataset_picked_directly` event. Both transition J-002 through
`switching_dataset_context` → `session_active` with
`active_scope.resource_*` populated AND
`session.active_dataset_id` persisted.

## Who

* **Maya Chen (in `session_active`, no dataset attached)** —
  references a dataset by name in a chat turn; the agent's
  `resolve_dataset` tool fires; she picks from the inline list.
  Wants the picked dataset to "stick" — the next turn should see
  it; resuming the session later should see it.
* **Maya Chen (in `session_active`, with a dataset attached)** —
  directly picks a different dataset from a list. Wants the
  same "stickiness" — the new dataset becomes the session's
  context.

## Solution

J-002's `session_active` state declares two transitions:

* `dataset_resolved_by_agent` → `switching_dataset_context` — fired
  by the FE in response to consuming a `data-agent-request` typed
  part (per `agent/lib/chat/handleChat.ts:99-104`). The event
  payload carries `intended_resource_type = "dataset"` and
  `intended_resource_id = <picked-id>`.
* `dataset_picked_directly` → `switching_dataset_context` — fired
  by the FE in response to a direct UI selection (e.g., from a
  dataset list in a side panel). Same event shape.

`switching_dataset_context` performs:

1. **Validate** the dataset id via ScopeResolver invariant 4
   (cross-tenant rejection per ADR-029 §1). If 403, transition
   back to `session_active` with `active_scope.resource_*` UNCHANGED
   and emit `dataset_access_denied` for FE feedback.
2. **Update** `active_scope.resource_type = "dataset"` and
   `active_scope.resource_id = <picked-id>`.
3. **Persist** `session.active_dataset_id = <picked-id>` to session
   metadata (per D11 / OQ-J002-1 storage shape).
4. **Transition** back to `session_active`.

The FE's re-submission of the chat turn (per
`features/chat-first-ui.feature:93`) carries the new
`X-Active-Scope` with `resource_type = "dataset"` and
`resource_id = <picked-id>` (per US-208's agent contract). The
agent dispatches the tool with the correct dataset id.

## Elevator Pitch

* **Before**: Maya types "filter rows where age > 30." Agent
  returns `resolve_dataset` tool call. FE shows picker. Maya picks
  "patients_2025". FE updates a TanStack Query key, a route param,
  and a React Context. The re-submission goes through. Some
  combination of those three pieces of state may have not synced;
  if Maya navigates away and back, the dataset chip may be gone.
* **After**: Maya types "filter rows where age > 30." Agent
  returns `resolve_dataset` tool call. FE shows picker. Maya picks
  "patients_2025". FE emits `dataset_resolved_by_agent` to J-002.
  J-002 transitions through `switching_dataset_context` →
  `session_active` with `active_scope.resource_id` populated AND
  `session.active_dataset_id` persisted. Re-submitted turn carries
  the right scope. Resuming the session later sees the dataset
  restored (per US-205).
* **Decision enabled**: Maya stops re-typing the dataset name. The
  resolution is "remembered" for the rest of the session AND
  restored on resume.

## Domain examples

### 1: Happy path — agent's `resolve_dataset` resolves to `patients_2025`

J-002 is in `session_active` for project Q4 Analytics with session
chat-9b2a and `active_scope.resource_* = null`. Maya types "filter
rows where age > 30." The chat turn dispatches; the agent's LLM
infers the prompt refers to a dataset and emits a
`resolve_dataset(name: "patients")` tool call. `pipeChatStream`
intercepts; the FE consumes the `data-agent-request` typed part and
renders an inline list: "patients_2025", "patients_archive". Maya
clicks "patients_2025". The FE emits
`dataset_resolved_by_agent { resource_type: "dataset", resource_id:
patients_2025_id }` to J-002. J-002 transitions:

* `session_active → switching_dataset_context`
* ScopeResolver validates access (✓).
* `active_scope.resource_type = "dataset"`, `resource_id =
  patients_2025_id`.
* `session.active_dataset_id = patients_2025_id` persisted.
* `switching_dataset_context → session_active`.

The FE re-submits the original chat turn "filter rows where age >
30" with the new `X-Active-Scope`. The agent dispatches
`filterTable` against `patients_2025` with the correct dataset id.

### 2: Happy path — direct selection from a dataset list

Maya is in `session_active` with `active_scope.resource_id =
sales_2026`. She wants to switch to `customers_2025`. She clicks
the dataset chip in the gutter; an inline list of project's
datasets appears; she clicks `customers_2025`. The FE emits
`dataset_picked_directly`. J-002 transitions through
`switching_dataset_context`; the agent's next chat-turn dispatches
to `customers_2025`.

### 3: Failure — picked dataset is cross-tenant (access denied)

Maya picks "restricted_dataset" (somehow visible in a list but
belongs to a different org). ScopeResolver invariant 4 rejects.
J-002 transitions back to `session_active` with the prior
`active_scope.resource_*` UNCHANGED. The FE shows inline copy "you
don't have access to that dataset" in the chat input gutter.

### 4: Failure — picked dataset belongs to a different project

Anomaly case (the dataset list should be filtered by project).
ScopeResolver detects mismatch (`dataset.project_id !=
active_scope.project_id`). 403 with named diagnostic. Same UX as
Example 3.

### 5: Edge — concurrent `dataset_resolved_by_agent` events

Two rapid-fire chat turns both surface `resolve_dataset` (a slow
user, slow agent, multi-turn scenario). The FE serializes — only
the most recent pick wins; the older event is dropped (XState's
state-machine semantics ensure only one `switching_dataset_context`
transition can run at a time).

### 6: Cross-flow — token expires during `switching_dataset_context`

(Covered by US-210.) FREEZE triggered mid-transition; the picked
dataset_id queues in the replay buffer; THAW replays.

## Acceptance Criteria

* [ ] J-002's `session_active` state declares both
  `dataset_resolved_by_agent` and `dataset_picked_directly` as
  transitions to `switching_dataset_context` with the same payload
  shape.
* [ ] `switching_dataset_context` calls ScopeResolver invariant 4
  (cross-tenant + cross-project rejection). On 403, transitions
  back to `session_active` with `active_scope.resource_*`
  UNCHANGED.
* [ ] On success, `active_scope.resource_type = "dataset"`,
  `active_scope.resource_id = <picked-id>`, AND
  `session.active_dataset_id = <picked-id>` persisted to session
  metadata (per D11).
* [ ] The chat turn re-submission after a successful resolution
  carries the new `X-Active-Scope` with `resource_*` populated
  (per US-208's contract).
* [ ] The FE intercepts `data-agent-request` typed parts (per
  `agent/lib/chat/handleChat.ts:99-104`) and emits
  `dataset_resolved_by_agent` to J-002. No other code path emits
  this event.
* [ ] The TS harness exposes:
  - `harness.j002.attach_dataset_via_agent(dataset_name)` —
    simulates the agent-tool-call path.
  - `harness.j002.attach_dataset_directly(dataset_id)` — direct
    selection path.
  - `harness.j002.assert_scope({resource_type, resource_id})`
    (already inherited).

## Outcome KPIs

* **Who**: Users in `session_active` who reference a dataset by
  name OR pick one from a list.
* **Does what**: Sees the dataset chip update; subsequent chat
  turns dispatch to the correct dataset; the resolution persists
  across session resume (composing with US-205).
* **By how much**: 100% of successful dataset resolutions persist
  to session metadata at p99; cross-tenant pick rejections
  surface the named-diagnostic copy at p99.
* **Measured by**: FE events
  `j002_dataset_resolution_completed { session_id, prior_resource_id,
  new_resource_id, path: "agent" | "direct", latency_ms }`,
  `j002_dataset_access_denied { reason: "cross_tenant" |
  "cross_project" }`.
* **Baseline**: Today, the FE-level state-management is the
  drift surface; cross-session persistence is zero (it's not
  built). Cross-tenant safety is partial.

## Technical notes

* Depends on US-205 (session-metadata storage shape OQ-J002-1
  resolved by DESIGN).
* Depends on US-208 (X-Active-Scope contract) so re-submitted
  turns carry the new scope correctly.
* The FE's interception of `data-agent-request` typed parts is
  ALREADY in place (per `agent/lib/chat/handleChat.ts:99-104`);
  what's new is the emit-to-J-002 event. The chat-view component's
  `data-agent-request` handler gains one line:
  `j002Machine.send({ type: 'dataset_resolved_by_agent', ...
  payload })`.
