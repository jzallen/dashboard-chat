# US-210: J-002 honors FREEZE/THAW from J-001's expired_token

> **Wave**: DISCUSS — `project-and-chat-session-management` (J-002)
> **Date**: 2026-05-13
> **Persona**: Maya Chen (mid-J-002-mutation when her JWT expires)
> **Job mapped**: J002-Job-8 (Survive a token expiry without
> re-submitting)
> **Slice**: Slice 6 — Cross-machine freeze/thaw
> **Anticipated machine state names**: `freeze`; transition target
> from `freeze` is the originating live state

## Problem

ADR-028 §"Decision outcome" specifies the cross-machine FREEZE/THAW
contract: when J-001 transitions to `expired_token`, the
orchestrator broadcasts FREEZE to every spawned child actor. Each
flow's machine declares a FREEZE handler that pauses outgoing
mutations. The orchestrator's replay buffer queues intent events
during the freeze window (5s timeout, 16 max per flow per ADR-027
§5). On THAW (silent re-auth success), queued intents replay with
their original `correlation_id`.

J-001 implements this end-to-end (its own `expired_token` side-state
is the canonical example). But **the cross-machine payoff is zero
with only one machine**. J-002 is the first machine that
**participates** in the FREEZE/THAW contract. This story validates
the substrate scales to a second flow — the architectural payoff
ADR-028 §94 names ("Cross-machine freeze (US-005) is a 5-line
`system.get(...).send(...)` loop, not a hand-rolled pub/sub").

Without this story, J-002's mutations during a token expiry would
either:
* Fail mid-flight with a 401, surfacing as a generic error (poor
  UX);
* Or worse, complete with stale credentials and then 401 on the
  next attempt — confusing the user about what actually happened.

## Who

* **Maya Chen (in any active J-002 state, JWT expires mid-mutation)** —
  whether mid-`creating_project`, mid-`loading_session_list`,
  mid-`resuming_session`, mid-`switching_project`, or
  mid-`switching_dataset_context`. Wants the action to NOT be lost.

## Solution

J-002's machine declares a top-level FREEZE handler reachable from
EVERY non-terminal state:

* From any state, on event `FREEZE`, transition to the `freeze`
  side-state, recording `last_live_state` in machine context.
* On event `THAW`, transition from `freeze` to `last_live_state`
  (XState history target).

In the `freeze` state:
* **No outgoing mutations** — J-002 emits no backend calls, no
  projection writes, no agent dispatches.
* **Intents queue at the orchestrator boundary**. The orchestrator's
  replay buffer captures `(intent_event, original_correlation_id,
  queued_at)` triples. On THAW, the orchestrator replays them.

The freeze trigger is purely orchestrator-driven. J-002 does NOT
import the J-001 machine (per ADR-028:46-48); it only knows about
the `FREEZE` event the orchestrator broadcasts. J-002 also does
not emit its own FREEZE (it's a downstream consumer, not an
upstream emitter).

## Elevator Pitch

* **Before** *(if J-002 had no FREEZE handler)*: Maya clicks
  "Q3 Sales" while her token is 1s from expiring. The switch
  fires; mid-load, the JWT expires; the backend returns 401; the
  switch fails with a generic error. Maya sees "Couldn't load Q3
  Sales" and re-clicks. The re-click also fails (her JWT is still
  expired, and silent re-auth hasn't run). She refreshes the page.
* **After**: Maya clicks "Q3 Sales" while her token is 1s from
  expiring. J-002 transitions to `switching_project`. The JWT
  expires; J-001 transitions to `expired_token`; orchestrator
  broadcasts FREEZE. J-002 enters `freeze` with
  `last_live_state = switching_project`. The intent is queued. FE
  shows "Refreshing your session..." banner. Silent re-auth
  completes; orchestrator broadcasts THAW. J-002 re-enters
  `switching_project` with the original correlation_id; the load
  completes; Maya lands in Q3 Sales.
* **Decision enabled**: Maya never re-clicks. Mid-task token expiry
  is invisible. The trust that "the app just works" extends from
  J-001 (chat turns) to J-002 (everything else).

## Domain examples

### 1: Happy path — token expires during session_clicked

J-002 is in `session_list_visible`. Maya clicks "chat-9b2a". J-002
transitions to `resuming_session`. The transcript-load call fires.
Mid-call, the JWT expires. The backend returns 401; J-001 detects
the expiry and transitions to `expired_token`; orchestrator
broadcasts FREEZE. J-002 transitions
`resuming_session → freeze` with `last_live_state = resuming_session`.
The transcript-load result (a 401) is discarded by J-002 (it does
not transition); the intent is preserved in the replay buffer with
the original correlation_id. The FE shows the "Refreshing your
session..." banner.

J-001's silent re-auth completes successfully; orchestrator
broadcasts THAW. J-002 transitions `freeze → resuming_session` with
the original correlation_id. The transcript-load fires again (this
time with a fresh JWT); succeeds; J-002 reaches `session_active`.

### 2: Happy path — token expires during switching_project

Same shape but `last_live_state = switching_project`. On THAW,
J-002 re-enters `switching_project`; the project-load fires with
the new JWT; J-002 reaches `project_selected` for the new project.

### 3: Edge — silent re-auth fails (J-001 transitions to error_recoverable instead)

J-001's silent re-auth fails. J-001 transitions to
`error_recoverable` (NOT `ready`). The orchestrator does NOT
broadcast THAW. J-002 stays in `freeze`. After the 5s timeout
(ADR-027 §5), the orchestrator emits `replay_abandoned`. J-002
transitions to `error_recoverable` carrying the original
correlation_id with copy "we couldn't complete that action — try
again."

### 4: Edge — multiple intents queued during a single freeze window

Maya clicks "Q3 Sales" AND clicks "chat-xyz" in rapid succession,
both during the freeze window. The orchestrator's replay buffer
holds both. On THAW, the orchestrator replays them in arrival
order. J-002 processes them serially: `switching_project` →
`project_selected` (for Q3 Sales); then the `session_clicked` for
chat-xyz fires (but chat-xyz is a Q4 session, not Q3, so it doesn't
match Q3's session list — J-002 emits a
`stale_intent_dropped_after_thaw` event for observability). The
final state is Q3 Sales' `session_list_visible`.

### 5: Edge — freeze during `session_active_no_messages` (no in-flight mutation)

Maya is in `session_active_no_messages` with the welcome chips
visible. Her token expires (J-001 expired_token from inactivity).
J-002 transitions to `freeze` with `last_live_state =
session_active_no_messages`. No mutation was in flight. On THAW,
J-002 returns to `session_active_no_messages`. From Maya's
perspective: the "Refreshing your session..." banner flickered;
the welcome state stayed visible the whole time.


## Acceptance Criteria

* [ ] J-002's machine declares a top-level FREEZE handler reachable
  from every non-terminal state (per the journey YAML).
* [ ] On FREEZE, J-002 transitions to `freeze` and records
  `last_live_state` in machine context; no outgoing mutations fire
  while in `freeze`.
* [ ] On THAW, J-002 transitions back to `last_live_state` with the
  original correlation_id preserved.
* [ ] Mid-flight backend responses arriving during `freeze` (e.g.,
  a 401 from a `list_sessions` call) are discarded by J-002 (no
  transition).
* [ ] The orchestrator's replay buffer captures J-002 intents
  arriving during `freeze` with their original correlation_ids;
  replays them in arrival order on THAW (per ADR-027 §5).
* [ ] Stale intents (intents whose target no longer applies after
  the THAW state — e.g., a `session_clicked` for a session in the
  prior project after a project switch happened during freeze) are
  dropped with a `stale_intent_dropped_after_thaw` observability
  event; J-002 does NOT raise an error to the user.
* [ ] On replay buffer timeout (5s per ADR-027 §5) without THAW,
  J-002 transitions `freeze → error_recoverable` with the original
  correlation_id and the originating user-action preserved in the
  failure event payload.
* [ ] The TS harness exposes `harness.j002.freeze()` and
  `harness.j002.thaw()` simulating the orchestrator broadcasts.

## Outcome KPIs

* **Who**: Users mid-J-002-mutation when their JWT expires.
* **Does what**: Mutation pauses; silent re-auth completes;
  mutation resumes; user never re-clicks or re-types.
* **By how much**: ≥95% of J-002-mid-mutation expiries are
  recovered silently (only ~5% require user-visible "Sign in
  again" surface — the silent-re-auth-failed case from US-005);
  0 re-clicks observed in instrumented sessions.
* **Measured by**: FE+UI-state events
  `j002_frozen { last_live_state, queued_intents_count }`,
  `j002_thawed { last_live_state, replayed_intents_count,
  stale_intents_dropped_count }`. Correlated with K4 from
  J-001's outcome-kpis.md (which extends from chat turns to J-002
  mutations under this story).
* **Baseline**: Today, no J-002 machine exists. The current FE's
  cross-flow freeze coordination is 0% (every mid-task expiry
  surfaces as a generic error). This story is the architectural
  payoff promised by ADR-028 §94.

## Technical notes

* Depends on US-201 through US-209 (J-002 has live states to
  freeze).
* The orchestrator's FREEZE/THAW broadcast is implemented in
  `ui-state/index.ts` from J-001's DELIVER. J-002 only adds a
  handler.
* The `stale_intent_dropped_after_thaw` filter rule is
  DESIGN-deferred via OQ-J002-6 in handoff (it asks: which
  queued intents are still valid after THAW given that J-002's
  state may have transitioned to a different sub-state during
  the freeze).
