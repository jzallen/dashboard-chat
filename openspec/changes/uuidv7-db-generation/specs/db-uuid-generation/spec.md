## ADDED Requirements

### Requirement: Database-level UUIDv7 generation via server_default
All ORM entity models SHALL use `server_default=text("uuidv7()")` on their `id` column instead of a Python-side `default=`. The generated DDL SHALL be valid for both PostgreSQL 18 (native function) and SQLite (custom function shim).

#### Scenario: Column DDL includes server default
- **WHEN** SQLAlchemy generates DDL for any entity table's `id` column
- **THEN** the DDL SHALL include a `DEFAULT` clause referencing `uuidv7()`
- **AND** on PostgreSQL the clause SHALL render as `DEFAULT uuidv7()`
- **AND** on SQLite the clause SHALL render as `DEFAULT (uuidv7())`

#### Scenario: PostgreSQL uses native uuidv7()
- **WHEN** a record is inserted into PostgreSQL 18 without an explicit `id` value
- **THEN** the database SHALL generate the ID using its built-in `uuidv7()` function
- **AND** the resulting ID SHALL be a valid UUIDv7 string in 36-character hyphenated lowercase format

#### Scenario: Explicit ID overrides server_default
- **WHEN** a record is inserted with an explicit `id` value
- **THEN** the explicit value SHALL be used regardless of dialect
- **AND** the `server_default` SHALL NOT fire

### Requirement: SQLite custom uuidv7() function for dev
A `uuidv7()` SQL function SHALL be registered on every SQLite connection via SQLAlchemy's `"connect"` event on `engine.sync_engine`. The function SHALL take zero arguments and return a valid UUIDv7 string.

#### Scenario: Dev engine registers uuidv7 on connect
- **WHEN** the application engine uses a SQLite database URL
- **THEN** a `"connect"` event listener SHALL register a `uuidv7` function (0 arguments) on the raw DBAPI connection
- **AND** the function SHALL delegate to `uuid_utils.uuid7()` and return `str(uuid7())`

#### Scenario: Function is available for DEFAULT evaluation
- **WHEN** SQLite evaluates a `DEFAULT (uuidv7())` clause during INSERT
- **THEN** the registered function SHALL be invoked and return a valid UUIDv7 string

#### Scenario: Function works with aiosqlite
- **WHEN** the async engine uses the `sqlite+aiosqlite` dialect
- **THEN** the event listener on `engine.sync_engine` SHALL correctly register the function on the underlying `sqlite3.Connection`

### Requirement: No Python-side UUID generation in ORM model defaults
All ORM record classes SHALL NOT use `default=lambda: str(uuid7())` or any other Python-side callable for the `id` column. The `uuid_utils` import SHALL be removed from all model files.

#### Scenario: ORM records have no default= parameter on id
- **WHEN** inspecting the `id` column definition of `ProjectRecord`, `DatasetRecord`, `TransformRecord`, `OrganizationRecord`, `ExternalAccessRecord`, or `OutboxRecord`
- **THEN** there SHALL be no `default=` parameter
- **AND** there SHALL be a `server_default=text("uuidv7()")` parameter

#### Scenario: No uuid_utils import in model files
- **WHEN** inspecting the import section of any ORM record file
- **THEN** there SHALL be no import of `uuid_utils` or `uuid7`

### Requirement: Use cases do not generate UUIDs in Python
The `create_dataset_from_upload` and `create_organization` use cases SHALL NOT import or call `uuid7()` or `uuid4()` for ID generation. IDs SHALL come from the database.

#### Scenario: create_dataset_from_upload pre-fetches ID from database
- **WHEN** `create_dataset_from_upload` needs an ID to compute `storage_path`
- **THEN** it SHALL obtain the ID by calling a repository method that executes `SELECT uuidv7()` on the current database session
- **AND** SHALL NOT call `uuid7()` or `uuid4()` in Python

#### Scenario: create_organization lets database generate org ID in dev mode
- **WHEN** `create_organization` runs in dev mode (not WorkOS)
- **THEN** it SHALL NOT call `uuid4()` to generate an org ID
- **AND** it SHALL let the database generate the ID via the ORM `server_default` when inserting the `OrganizationRecord`

### Requirement: Repository exposes ID generation method
The metadata repository SHALL expose a method for pre-fetching a database-generated UUIDv7 ID.

#### Scenario: generate_id returns a valid UUIDv7
- **WHEN** a use case calls `metadata_repo.generate_id()`
- **THEN** it SHALL return a string containing a valid UUIDv7
- **AND** the value SHALL have been generated by executing SQL against the current database session

### Requirement: Migration squash to single initial schema
All existing Alembic migrations (001–013) SHALL be replaced with a single `001_initial_schema.py` that creates all tables from the current ORM model definitions, including `server_default=text("uuidv7()")` on ID columns.

#### Scenario: Single migration creates complete schema
- **WHEN** `alembic upgrade head` runs against an empty database
- **THEN** a single migration SHALL create all tables matching the current ORM model definitions
- **AND** all ID columns SHALL have `server_default` set to `uuidv7()`

#### Scenario: No intermediate migration history
- **WHEN** inspecting the `backend/migrations/versions/` directory
- **THEN** there SHALL be exactly one migration file (plus `.gitkeep`)

### Requirement: PostgreSQL 18 in Docker Compose
The `db` service in `docker-compose.yml` SHALL use `postgres:18` (or `postgres:18-alpine` when available).

#### Scenario: PostgreSQL version supports native uuidv7
- **WHEN** the `db` Docker service starts
- **THEN** it SHALL run PostgreSQL 18 or later
- **AND** the `uuidv7()` function SHALL be available without any extension installation
