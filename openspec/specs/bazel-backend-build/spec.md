## Purpose

Defines the Bazel build surface for the Python backend — library targets, hermetic pytest execution from `uv.lock`, and the layered OCI image used in production. It is the backend's entry point into the Bazel monorepo build graph.

## Requirements

### Requirement: Backend Python library target

The backend SHALL expose a `py_library(name = "app")` target in `backend/` that encompasses all source under `backend/app/` and resolves its dependencies from the `@pip//` repository generated by `rules_uv` from `uv.lock`.

#### Scenario: Library target builds backend sources

- **WHEN** `bazel build //backend:app` is run
- **THEN** Bazel SHALL compile and package all files under `backend/app/`
- **AND** dependencies SHALL resolve from `@pip//` generated by `rules_uv` from `backend/uv.lock`

### Requirement: Backend test target

The backend SHALL expose a `py_test(name = "tests")` target that runs pytest under Bazel's sandbox with the project's fixtures, dev dependencies, and parallelism settings.

- Test inputs SHALL include all files in `backend/tests/` and `conftest.py` at each directory level.
- Dev dependencies (`pytest`, `pytest-asyncio`, `pytest-xdist`, `moto`) SHALL be available to the test target.
- Pytest options SHALL include `asyncio_mode = "auto"` and `-n auto --dist=loadfile`.
- The test target SHALL include the `mock_s3` fixture (moto) and a temporary SQLite database.

#### Scenario: Running backend tests via Bazel

- **WHEN** `bazel test //backend:tests` is run
- **THEN** pytest SHALL execute all tests under `backend/tests/` in a sandbox
- **AND** tests SHALL run in parallel with `-n auto --dist=loadfile`
- **AND** async tests SHALL run without explicit `@pytest.mark.asyncio` annotations because of `asyncio_mode = "auto"`

#### Scenario: S3 and SQLite fixtures available in tests

- **GIVEN** a test that relies on S3 or SQLite
- **WHEN** the Bazel test target runs
- **THEN** the `mock_s3` fixture from moto SHALL be available
- **AND** a temporary SQLite database SHALL be provided so tests do not require an external database

### Requirement: Backend OCI image

The backend SHALL publish an `oci_image(name = "image")` target producing a layered image with a pinned Python 3.11 base, system packages, pip dependencies, and application code, plus an `oci_tarball(name = "image.tar")` target for Docker loading.

- Base layer SHALL be `python:3.11-slim`.
- A system layer SHALL install `build-essential`, `libpq-dev`, and `curl`.
- A dependency layer SHALL contain pip packages resolved from `uv.lock`.
- An application layer SHALL include `app/`, `scripts/`, and `migrations/`.
- The image entrypoint SHALL be `["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]`.
- `scripts/setup_dev.py` SHALL be included for container startup seeding.
- Alembic migrations SHALL be bundled in the image but SHALL NOT be executed at build time.

#### Scenario: Image is layered for cache efficiency

- **WHEN** the backend image is built via Bazel
- **THEN** the resulting `oci_image` SHALL contain distinct layers for the Python base, system packages, pip dependencies, and application code
- **AND** changing application source SHALL invalidate only the application layer

#### Scenario: Image loadable into Docker via tarball

- **WHEN** `bazel build //backend:image.tar` completes
- **THEN** the resulting tarball SHALL be loadable via `docker load < bazel-bin/backend/image.tar`
- **AND** the loaded image SHALL start `uvicorn` on port 8000

### Requirement: Python version compatibility

The backend build SHALL use a Python version matching the `>=3.11, <3.13` constraint declared in `backend/pyproject.toml`.

#### Scenario: Python version matches pyproject.toml

- **WHEN** the backend's Bazel-registered Python toolchain resolves
- **THEN** the selected interpreter version SHALL satisfy `>=3.11, <3.13`
- **AND** mismatches SHALL surface as a build error, not a silent downgrade
