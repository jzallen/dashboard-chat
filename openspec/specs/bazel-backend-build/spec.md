## Purpose

Defines the Bazel build surface for the Python backend — library targets, hermetic pytest execution from `uv.lock`, and the layered OCI image used in production. It is the backend's entry point into the Bazel monorepo build graph.

## Capability: bazel-backend-build

BUILD files for the Python backend service — library targets, test targets, and OCI image.

### Behavior

- `py_library(name = "app")` in `backend/` encompasses all source under `backend/app/`
- Dependencies resolved from `@pip//` repository generated by `rules_uv` from `uv.lock`
- `py_test(name = "tests")` runs `uv run pytest` (or pytest directly) in sandbox with:
  - All test files from `backend/tests/`
  - `conftest.py` fixtures at each directory level
  - Dev dependencies (pytest, pytest-asyncio, pytest-xdist, moto)
  - `asyncio_mode = "auto"` via pytest.ini options
  - `-n auto --dist=loadfile` for parallel execution
- `oci_image(name = "image")` produces a layered image:
  - Base: `python:3.11-slim`
  - Layer: system packages (build-essential, libpq-dev, curl)
  - Layer: pip packages from uv.lock
  - Layer: application code (`app/`, `scripts/`, `migrations/`)
  - Entrypoint: `["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]`
- `oci_tarball(name = "image.tar")` for loading into Docker

### Constraints

- Python version must match `>=3.11, <3.13` from pyproject.toml
- Test target must include `mock_s3` fixture (moto) and temp SQLite database
- Alembic migrations included in OCI image but not executed at build time
- `scripts/setup_dev.py` included for container startup seeding
